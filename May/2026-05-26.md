# 데이터베이스
자료를 정보로 활용하기 위해서 구조화 해놓은 데이터 모음

## 데이터베이스가 없었다면?
파일 시스템으로 데이터를 관리한다면?

- 데이터 기록 중복 -> 중앙 집중 관리
- 파일 커지면 찾기 힘듬 -> 효울적 검색
- 보안, 권한 부족 -> 높은 보안성
- 동시에 수정 불가 -> 동시성 제어
- 데이터가 잘못 적혀있을 수도 있음 -> 무결성 제약
- 다른 os에서 제대로 안 읽힘 -> 호환성

## 관계형 데이터베이스의 등장
- 데이터를 표 단위로 관리
- 표 간의 관계를 표현하기

database : 구조화된 데이터 모음
relational database : 관계형 데이터 모음
rdbms : 관계형 데이터 모음 관리 시스템

## 테이블 설계하기
1. 엔티티 정의
   - 엔티티 : 데이터베이스에 넣고자 하는 대상
2. 속성 정의
   - 속성 : 대상이 가진 정보
3. 관계 정의

## SQL
- DDL : 데이터 정의어
- DML : 데이터 조작어
- DCL : 데이터 제어어

## ERD
- 개체와 관계를 한 눈에 알아볼 수 있도록 그려놓은 도표

1. 개념적 모델링 : 개체와 개체가 어떤 관계를 가지고 있는지 그림
2. 논리적 모델링 : 테이블에 어떤 속성이 있는지 -> ERD
3. 물리적 모델링 : 데이터베이스에 구현

식별 : 자식 테이블은 부모가 없으면 존재할 수 없을 때
비식별 : 연결된 테이블이 스스로 식별 가능할 때

현업에서는 주로 어떤걸 쓰냐?
- 비식별을 주로 씀
- 식별을 쓰면 무조건 join을 해야해서 성능 이슈가 있을 수 있다.

## 인덱스
- 테이블의 검색 속도를 높이기 위해 별도의 저장 공간에 데이터의 위치와 함께 정렬해 둔 자료 구조

- 해시 인덱스
  - 입력 값에 따라 같은 출력을 가짐
  - 탐색 시간이 제일 빠름
  - 범위 검색이 힘듬
  - 해시값 충돌 해결이 어려움

- 트리 인덱스
  - 노드로 이뤄진 자료 구조
    - 계층적인 데이터 구조
    - 크기 비교를 통해 값을 찾음
    - 한쪽으로만 정렬된 트리가 생길 수 있음 (우편향 트리)
    - 모든 노드를 거쳐야해서 느려짐

- B-Tree
  - 트리의 노드 깊이가 균형 있게 유지됨
  - 다양한 쿼리에 효율적으로 작동
  - 균형을 맞추기 위한 노드 분할, 병합 필요
  - 범위 검색 어려움

- B+Tree
  - 데이터를 리프노드에만 값을 저장하는 트리
  - 내부 노드는 탐색에만 사용
  - 내부 노드에 더 많은 키를 담을 수 있어 디스크 접근이 줄어든다.
  - 범위 검색 유리
  - 순차 접근 쉬움

## B-Tree 와 B+Tree

- 디스크에 있는 정보를 가져올때 페이지 단위로 들고 온다. (16kb)
- B-Tree (예시)
  - 노드에 저장한 값이
    - 키 값 -> 1kb
    - 주소값 -> 3kb
  - 두 정보를 같이 저장함 -> 4kb 
  - 데이터를 읽어올 때,
    - 16kb 페이지에 노드를 가져온다라고 하면
    - 4개의 노드만 가져올 수 있다.
- B+Tree
  - 리프 노드에만 주소 값을 가지고 있고
  - 다른 노드들에는 키값만 갖고 있다.
  - 한 개의 중간 노드에 저장되는 것은
    - 키 값 -> 1kb
    - 16kb 페이지라면 16개의 키를 한 번에 들고 올 수 있다.

## B+Tree
- 한번에 많은 키를 들고 올 수 있게 하고
- 리프 노드에 데이터를 몰아넣어서, 인접하는 노드를 링크드 리스트로 연결해두었다.

## Full Text Index
텍스트로 구성된 문자열 데이터의 내용을 가지고 생성한 인덱스

### 파서
1. stopword
   - 공백, 문장 부호 단어를 분리한다.
   - 영어권에서는 잘 먹힌다.
2. n-gram 파서
   - 지정된 글자 수 만큼 자르는 파서
   - 조사가 있던 말던 다 자르기 때문에
   - 한국어 같은 언어에서 stopword 파서 보다 검색 기능 성능이 좋다.
3. MeCab
   - 형태소 분석기
   - 한국어 검색에서 노이즈 준다.
   - 동의어 처리 문제
   - 문맥 분석 문제

### 한국어 검색은 성능을 높이려면 검색 엔진을 별도로 사용하는게 좋다.
- Elastic Search 별도의 검색 엔진을 두고 데이터를 밀어넣어서 검색은 별도 엔진에서 수행하는 것이 좋다.

- 기본적으로 무겁다.
- 전문 검색엔진이 아니다.

# 트랜잭션
- 데이터베이스의 상호작용의 단위

## 단위
- Active
- Partially Committed
  - SQL 명령이 성공적으로 완료됨, DB에 영구기록 전
- Committed
  - 트랜잭션 완료
  - DB 반영 완료
- Failed
  - 트랜잭션 실패
  - 트랜잭션 이전 상태로 돌아감 (Rollback)
- Aborted

## 트랜잭션 사용하는 이유
- 무결성과 일관성을 보장
- 여러 작업을 하나의 논리적인 작업 단위로 묶어 처리하기 위해

### 무결성
- 데이터에 결함이 없는 상태
- 데이터베이스에 저장된 데이터가 정확하다, 유효하다, 신뢰성이 있다.

### 일관성
- 모순이 없다. 규칙을 준수한다.
- 작업 전후에 상태가 일관적이다.

## ACID
- 트랜잭션이 안전하게 수행된다는 것을 보장하기 위한 원칙
  
### Atomicity
- 트랜잭션이 완벽하게 수행되거나 전혀 수행되지 않아야한다.

### Consistency
- 트랜잭션의 작업 처리 결과는 항상 일관성과 정합성이 보장되어야한다.

### Isolation
- 동시에 다수 쿼리가 발생하는 경우 모든 쿼리 당 트랜잭션은 독립 되어야 한다.

### Durability
- 트랜잭션이 성공적으로 완료되었으면, 결과는 영구적으로 반영되어야 한다.
