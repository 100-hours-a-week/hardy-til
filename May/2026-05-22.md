# 2. Web Server와 Web Application Server가 요청 처리 책임을 분리하는 구조를 분석하시오.

# Web Server

**웹 서버**는 “클라이언트가 요청한 **정적 콘텐츠**를 HTTP 프로토콜을 통해 제공하는 서버”이다.

**정적 콘텐츠**는 “서버에 저장되어 있으며 모든 사용자에게 동일하게 전달되는 파일”을 말한다.
예를 들면 html 파일, 이미지, 영상 등이 있다.

웹서버는 크게 두 가지 기능을 가지고 있다. **정적인 컨텐츠를 제공하는 기능**과 **동적인 컨텐츠 제공을 위한 전달하는 기능**이 있다.

최초의 웹 서버는 어떤 기능을 가지고 있을까? 최초의 웹 서버는 1990년 CERN 연구에서 팀버너스리가 만든 **CERN httpd**이다. CERN httpd의 소개 홈페이지에서 해당 프로그램이 어떤 기능을 가지고 있는지 살펴 볼 수 있다.

그 당시에도 정적 컨텐츠를 제공하는 역할 말고도 **프록시 서버, 동적 컨텐츠 제공, 캐싱** 등 다양한 기능을 제공하는 것을 알 수 있다.

웹 서버는 시간이 지남에 따라 **리버스 프록시, 로드 벨런서** 등등 기능이 더 추가되었다.

대표적인 웹 서버로는 **Apache HTTP Server, Nginx**가 있다.

### 정방향 프록시

- 클라이언트와 서버 간의 통신을 중계하는 역할을 하는 서버나 소프트웨어이다.
- 클라이언트 - 프록시 - 인터넷 - 서버

### 리버스 프록시

- 클라이언트의 요청을 대신 받아 내부 웹 서버에 전달하고, 서버의 응답을 클라이언트에게 반환하는 서버 앞단의 중개자이다.
- 클라이언트 - 인터넷 - 프록시 - 서버

### 차이

- 프록시가 어디에 위치해 있냐는 차이이다.

## 웹 서버의 동적 컨텐츠 핸들링

### CGI

웹 서버는 동적 컨텐츠를 제공하지 못하는 것 아닌가 생각이 들 수 있다. 하지만 웹 서버도 동적 컨텐츠를 처리할 수 있는 방법이 있다. 바로 동적 컨텐츠를 처리할 수 있는 곳으로 처리를 넘긴 후 결과를 클라이언트에게 반환하면 된다.

**CGI**는 웹 서버 상에서 프로그램을 동작시키기 위한 방법을 정의한 인터페이스이다. 이 인터페이스를 규칙에 따라 웹 서버와 상호작용하는 프로그램 코드를 **CGI 스크립트**라고 한다. CGI 스크립트는 **PHP,** **C, Perl, Python** 등 다양한 언어로 구현할 수 있다.

CGI 처리 흐름

1. 동적 콘텐츠가 필요한 경로로 요청이 들어온다면 웹 서버는 그 요청을 CGI 프로그램에 넘긴다.
2. CGI 프로그램에서는 요청에 대한 결과를 동적으로 생성하여 HTML을 만든 후 웹 서버에게 응답한다.
3. 이 응답을 그대로 사용자에게 반환한다.

CGI 프로그램은 어떤 문제가 있었을까? CGI 프로그램이 프로세스 단위로 실행되어 사용자가 많아지면 서버에 부하가 많이 간다.

# Web Application Server, WAS

**WAS**는 웹 서버로부터 오는 동적인 요청을 처리하는 서버를 의미한다. 웹 서버와 컨테이너를 붙여놓은 서버라고 볼 수 있다.

**WAS**는 웹 컨테이너 혹은 서블릿 컨테이너라고도 불림 **컨테이너**란 JSP, Servlet을 실행시킬 수 있는 소프트웨어이다. 즉, WAS는 **JSP, Servlet** 구동 환경을 제공한다.

CGI의 경우 대규모 시스템 설계에 적합하지 않고, 성능 문제가 있었다. 요청이 도착할때마다 CGI 프로세스를 가동하기 때문이다.

그때 자바의 Servlet이 이 문제를 해결한다. 원래 자바는 웹 어플리케이션을 위해 개발된 언어는 아니다. 그래서 JavaEE(Java Enterprise Edition)의 일부로서 **Servlet**이라는 웹 어플리케이션 개발을 지원하기 위한 기능을 제공했다.

**Servlet은** 자바로 만들어진, HTML 등의 웹 콘텐츠를 생성하기 위한 프로그램이다.

언어가 자바여서 대규모 어플리케이션 개발에 적합하다. 또한 웹 서버와 같은 프로세스 속에서 콘텐츠를 생성하는 프로그램이 작동하기 때문에 CGI처럼 새로운 프로세스를 매번 실행할 필요가 없어서, 비교적 빠른 속도를 갖추었다.

**Servlet Container**는 서블릿을 위한 컨테이너로, 서블릿의 생명주기를 관리하고, URL과 특정 서블릿을 맵핑하며, URL 요청이 올바른 접근 권한을 갖도록 보장한다. 서블릿의 표준은 인터페이스로 제공되며, 구현체는 서블릿 컨테이너인 **Tomcat, Jetty, Undertow**가 제공한다.

# Web Server 단독

- 동적 컨텐츠를 처리하는데 많은 자원이든다.

# WAS 단독

- 정적 컨텐츠 서빙과 동적 컨텐츠 처리 역할을 동시에 맞게 되어 서버에 부하가 가게 된다.

# Web Server와 WAS 조합

1. **책임 분할**
    1. 책임 분할을 통해 서버 부하를 방지할 수 있다.
    2. 정적 컨텐츠는 웹 서버가, 동적 컨텐츠는 WAS가 담당하도록 하여 WAS에게 몰리는 서버 부하를 분산할 수 있다.
2. **로드 밸런싱**
    1. WAS에 가해지는 부하를 분산 시켜줄 수 있다.
    2. 웹 서버가 오류가 발생한 WAS로는 요청을 보내지 않고 다른 WAS로 요청을 보내게 할 수 있다.
3. **SSL 암복호화 처리에 Web Server 사용 (TLS/SSL 오프로딩)**

### 로드 밸런싱

- 네트워크 또는 서버에 가해지는 부하 트래픽을 분산 시켜주는 기술이다.

### TLS/SSL 오프로딩

- 프록시 혹은 로드 밸런스 단에서 암호화된 TLS/SSL 트래픽 해독 작업을 진행하여 각 애플리케이션에서 암호 해독을 수행할 때 드는 비용이 제거된다.
- SSL 인증서를 각 애플리케이션마다 할 필요가 없어진다.
- 내부적으로는 HTTP를 통해 소통할 수 있어서 빠른 속도로 통신이 가능하다.
