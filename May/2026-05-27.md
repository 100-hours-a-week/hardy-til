# HTTP 자격 증명 헤더
- HTTP는 이전 요청에 대해 기억하지 않는다.
- 접근할때마다 누구인지 증명해야한다.
- 헤더를 통해 인증한다.

## WWW-Authenticate
- 서버가 발급하는 Response Header
- 클라이언트의 요청이 인증 필요하다고 알려줌

## Authorization
- 클라이언트가 서버한테 요청할 때 보내는 헤더

## HTTP 인증 헤더 방식
- Basic Authentication
  - 기본 인증
  - 아이디 + 패스워드 -> Base64 인코딩
  - Authorization: Basic asdfsdf=

- Digest Authentication
  - 거의 사장됨

- Bearer Token
  - 기존 문제
    - Basic -> 비밀번호 매번 평문 보내기, 매 요청마다 비번 체크
    - Digest -> 비밀번호 암호화 -> 매 요청마다 서버가 체크
  
  - 이 토큰을 소지한 요청은 인증, 인가 된걸로 한다.
  - 유저의 아이디 비번은 한 번만 전송된다. 
    - 로그인 시도
    - 서버 인증 수행
    - 토큰 발급
    - 요청마다 토큰 포함
  - 이제부터 서버는 데이터베이스에 비밀번호 조회를 한 번만 하면 된다.

  - 장점
    - 유저의 비밀번호 보호
    - 범용성 좋음
    - CSRF 방어됨
  - 단점
    - 토큰 탈취당하면 위험하다
      - 토큰 탈취가 상당히 쉽다
    - 토큰은 보관하기 까다롭다
      - 브라우저 저장 공간은 2개
        - storage -> 자바스크립트로 값을 가져올 수 있음 -> XSS 스크립트 공격 취약
        - cookie -> 항상 모든 요청에 쿠키가 자동으로 붙음 -> CSRF 취약
  - 한계
    - 오페이크 토큰, 불투명 토큰
      - 토큰이 값이 의미가 없는 토큰
    - 서버가 보관하고 조회하는 자원을 사용해야함
  - 서버가 DB를 안뒤져도, 토큰 자체에 유저 정보가 있고 위조 되었는지 스스로 증명할 수 있으면 서버 자원을 안 써도 되겠네
  - JWT

# JWT (JSON Web Token)
- 당사자 간에 정보를 JSON 객체 형태로 안전하고 가볍게 전송하기 위한 웹 표준 개방형 규격

## 구조
- header . payload . signature
- header, payload -> base64
- signature -> 헤더와 페이로드 합쳐서 해시하고 비밀키를 사용하여 서명한 결과 

## 장점
1. 완전한 무상태성 + 확장성
2. DB 조회를 상당히 적게한다.

## 단점
1. 무효화가 어렵다.
2. 네트워크 대역폭이 낭비된다.

## RTR (Refresh Token Rotation)
리프레시 토큰 재사용하지 말고, 한번 쓰면 폐기하고 다시 새걸로 바꾸자

1. 유저의 액세스 토큰(A1)이 만료됨
2. 유저가 리프레스 토큰(R1)을 서버에 보내면서 엑세스 토큰 다시 만들어줘
3. 서버는 본인의 저장소 (Redis) R1이 유효한지 확인. R1을 폐기
4. 서버는 새로운 A2, R2 발급
5. 서버는 R2를 저장소에 저장

### 상황
1. 유저의 리프레시 토큰이 해커한테 탈취 당함
2. 해커가 탈취한 리프레시 토큰으로 액세스 토큰을 요청함
3. 서버는 저장되어 있던 리프레시 토큰을 없애고, 다시 액세스 토큰과 리프레시 토큰을 제공함
4. 유저의 액세스 토큰이 만료되어 리프레시 토큰으로 액세스 토큰을 요청함
5. 리프레시 토큰은 유효한데 서버에 없음 -> 리프레시 토큰 탈취 정황 감지
6. 서버에서 해당 유저의 리프레시 토큰을 모두 만료 시킴

그러면 매번 DB 조회하지 말고 한번만 인증한 다음에
인증했다는 증표를 주면 되지 않을까?
-> Bear Token

근데 이 토큰값은 아무런 의미가 없이 키로만 쓰이는데 토큰값에 유저 정보를 담고 이를 위조 방지되었는지 확인할 수 있으면 좋지 않을까?
-> JWT

JWT 구조는 header, payload, signature
header에 서명에 사용된 알고리즘 종류
payload에 유저 정보
signature에 인코딩한 header와 payload, 시크릿 키를 결합해 서명 생성

서버는 서명을 재계산해서 수신된 서명과 일치하는지 확인하여 토큰 위변조 여부 판단
