서블릿
- 동적 웹페이지를 만들 때 사용되는 자바 기반의 웹 애플리케이션 프로그래밍 기술
- 클라이언트 요청을 해석, 데이터 파싱, 응답을 생성해서 다시 전달하는 역할
- 서블릿을 사용해서 개발자는 비즈니스 로직에 집중할 수 있음

서블릿 컨테이너
- 서블릿의 생명 주기 관리

프론트 컨트롤러 패턴
- 컨트롤러마다 다 서블릿을 연결해줘야하는데, 이걸 공통 로직으로 묶어서 처리 -> DispatcherServlet이 이걸 담당
- 근데 이러면 너무 바빠지니깐 역할을 분리
- 요청 -> 요청 처리 핸들러 검색(Handler Mapping) -> 핸들러 호출(Handler Adapter) -> 뷰 검색 또는 생성(View Resolver) -> 응답
  - 이게 MVC 패턴이다

MVC
- 모델 뷰 컨트롤러로 역할을 나누는 패턴
- 코드의 복잡성을 줄이고 유지보수성을 높임
  - 컨트롤러
    - 사용자 요청을 URL 단위로 받음
    - 모델 뷰 연결
  - 모델
    - 비즈니스 로직, 데이터
  - 뷰
    - 사용자가 보는 화면
    - JSON

Spring MVC
- 웹 요청을 Controller가 받아 제어하고, Model과 View를 연결해 응답을 생성하도록 설계된 Spring 기반 웹 MVC 프레임워크

면접관은 신입에 어떤걸 바랄까?
- 성장 가능성을 본다
- 원리를 알려고 하는지?
  - 가시성이 낮은거에 대해서 원리를 알고 있냐
    - HTTPS 키교환 방식에 대해 설명해봐라
    - Dispatcher Servlet 작동 방법

HTTPS
- HTTP + Secure
- 암호화
  - 대칭키
    - 암호화, 복호화를 1개의 키로
    - 장점 : 데이터 전달이 빠름
    - 단점 : 키를 전달하기 어려움
  - 비대칭키
    - 암호화는 공개키로만
    - 복호화는 비공개키로만
    - 장점 : 키 교환 안전
    - 단점 : 대칭키보다 처리가 느림
- 클라이언트의 대칭키를 안전하게 주고 받는 과정

전자서명
- 개인키로 암호화하고 공개키로 복호화함

인증
- 통신하는 상대방이 누군지 신원 확인

1. 서버 측에서 인증서를 만듦
   1. 서버 -> 인증기관 : 서버 공개키, 인증 정보
   2. 인증기관 -> 서버 : 인증서 발급 -> 서버 공개키, 인증 기관 정보, 인증 기관 전자 서명 (인증 기관 비밀키로 서명)
2. 클라이언트가 서버측에 HTTPS 통신을 위해 인증서 요청
3. 받은 인증서를 브라우저 내부에서 검증 (인증 기관 목록, 인증 기관 공개키)
4. 클라이언트가 만든 대칭키를 인증 기관 공개키로 암호화하고 서버로 보냄
5. 서버는 대칭키를 비밀키로 복호화함
