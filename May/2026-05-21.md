# 상태와 연결

statefull
- 상태를 기억하는 것
- ex) 카페 점원이 고객을 기억함
- 장점
  - 맞춤형 서비스 가능
  - 복잡한 흐름 제어 가능
  - 게임, 채팅 등에서 사용
- 단점
  - 서버가 상태를 계속 기억해야함
  - 사용자가 많아지면 부담
  - 분산 시스템에서 처리 힘듬

stateless
- 상태를 기억하지 않는 것
- ex) 카페 점원이 고객을 기억하지 않음
- 장점
  - 서버 부담이 적음
  - 확장하기 편리함
  - 처리 속도가 빠름
- 단점
  - 복잡한 요청에 보낼 정보 많음
  - 사용자별 서비스 어려움
  - 순서가 있는 경우 관리 어려움

connection oriented
- 계속 연결을 유지
- 실시간성을 보장해야할때 사용
  - 장점
    - 계속해서 통신을 할 수 있음
  - 단점
    - 한 서버에서 동시에 연결을 유지할 수 있는 사용자가 적음

connectionless
- 데이터를 주고 받을때만 연결을 맺고, 바로 연결을 끊음
- 장점
  - 동시에 많은 사용자를 대응할 수 있다.
- 단점
  - 데이터를 주고 받을때마다 연결을 해야한다.

인증
- 사용자의 신원을 확인하는 보안 절차

인가
- 사용자와 제작사 사이에 프로그램 사용 권한을 주고 받는 계약

쿠키
- 서버를 통해 인터넷 사용자의 컴퓨터에 설치되는 작은 기록 파일
- 웹 사이트의 방문 정보를 기억해 개인화된 서비스를 제공하기 위해서
- 노출해도 되는 것들을 담는다
  - 방문 기록
  - 장바구니에 담은 것들
  - 언어 설정
- 서버에 요청할때 쿠키가 자동으로 붙는다.
- 단점
  - 보안에 취약함

세션
- 사용자의 정보를 서버측에서 관리해 보안을 강화하고 사용자 별 맞춤형 정보를 제공하는 것
- 사용자 인증 후 사용자 정보를 session 저장소에 넣고 이를 조회할 수 있는 session id를 쿠키에 담아줌
- 사용자가 session id를 가지고 요청하면 session 저장소에서 사용자 정보를 조회해서 로직을 처리함

프레임워크
- 개발을 위한 구조와 규칙을 제공
  - 개발 과정 단순화
  - 개발 속도를 높임
  - 여러 명이 작업해도 일관적
- 일관성 제공

라이브러리
- 개발자가 필요한 기능을 호출해서 사용하도록
- 개발자 혼자 만들기 힘든걸 가져와서 씀
  - 장점
    - 빠른 개발
    - 높은 완성도
- 전문성 제공

프레임워크, 라이브러리 차이
- 규제와 규약 -> 프레임워크
- 자율성 -> 라이브러리

API
- 서로 다른 소프트웨어 시스템이 통신하고 데이터를 교환할 수 있게 해주는 연결 고리

찰리의 규칙
- 코드는 가독성이 높아야한다.
- 코드의 의도를 주석으로 상세 기록해야한다.

페이징
- 한정된 용량에서 데이터를 한 번에 처리할 수 있게 적당한 크기로 나눠서 페이지로 처리하는 것
- 표현 방식
  - 페이지
  - 인피니티 스크롤
- 구현 방식
  - 오프셋 베이스
    - offset -> 건너띄는 양
    - limit -> 가져올 양
    - 단점
      - offset이 커지면 그만큼 읽고 버려야하기 때문에 성능 저하 발생
      - 중복이 생김
        - offset 구간에 새로운 데이터가 들어오면 새로운 데이터를 포함해서 offset 카운트를 하기 때문에 중복된 데이터가 들어갈 수 있음
  - 키 베이스
    - 키 -> 데이터의 키(인덱스) 값을 기준으로 데이터 가져옴
    - 장점
      - 큰 데이터에서도 조회 속도가 빠르다
      - 데이터가 추가 삭제 되어도 안정적임
    - 단점
      - 특정 구간으로 이동 힘듬
      - 순차 탐색만 가능
      - 키로 등록해야만 정렬 가능
    - 무한 스크롤, 인피니티 스크롤에 사용
